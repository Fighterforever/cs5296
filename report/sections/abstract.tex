\begin{abstract}
\noindent
AWS exposes the same workload to three very different compute paradigms ---
virtual machines on EC2, managed containers on ECS Fargate, and serverless
functions on Lambda. Selecting between them is rarely a clean choice because
performance, elasticity, and cost rarely move in the same direction. We
deploy an identical container image of a URL-shortener service backed by
DynamoDB to all three paradigms behind a single Application Load Balancer,
and benchmark them against four scenarios: a steady-state RPS sweep up to
500\,RPS, a step-arrival burst, a cold-start probe, and a mixed read/write
workload. To keep the comparison fair we use the AWS Lambda Web Adapter so
that the exact same FastAPI binary runs on every paradigm, and we drive load
from a co-located \texttt{c5.large} k6 worker. Across our measurements EC2
delivers the lowest steady-state tail latency, Fargate offers the best
balance of elasticity and operational cost, and Lambda absorbs bursts within
a few seconds at the price of a roughly $\textbf{40$\times$}$ cold-start
penalty on the first request after idle. We then derive analytic
break-even points by combining the measured per-request latency with public
April~2026 us-east-1 pricing: Lambda dominates Fargate up to roughly
\textbf{17\,RPS} of sustained traffic and Fargate dominates EC2 up to about
\textbf{60\,RPS} for a 1\,GB workload. Our infrastructure is fully
described in Terraform and runs unmodified inside the AWS Academy Learner
Lab; the artifact, k6 scripts, and analysis pipeline reproduce every figure
from raw data with a single \texttt{make} target.
\end{abstract}
