\begin{abstract}
\noindent
AWS exposes the same workload to three very different compute paradigms:
virtual machines on EC2, managed containers on ECS Fargate, and serverless
functions on Lambda. Choosing among them is rarely clean because
performance, elasticity, and cost pull in different directions. We
deploy an identical container image of a URL-shortener backed by
DynamoDB to all three paradigms behind a single Application Load
Balancer, and benchmark them under four scenarios: a steady-state RPS
sweep, a step-arrival burst, a cold-start probe, and a mixed
read/write workload. To keep the comparison fair we use the AWS Lambda
Web Adapter so the same FastAPI binary runs on every paradigm, and
drive load from a co-located k6 worker in the same availability zone.
Within a new AWS account's default service quotas, Lambda delivered
the most stable latency (p95~$\approx$~40\,ms, p99~$\approx$~52\,ms across
25--75\,RPS); a single-instance EC2~ASG held a 13\,ms median to 200\,RPS
before its burstable-credit tail blew out; and a single 0.5\,vCPU
Fargate task saturated at 100\,RPS with p50 above 3\,s. Combining
these with public April~2026 us-east-1 pricing, we derive a single
break-even crossover near \textbf{20\,RPS} --- below it Lambda's no-idle
billing wins, above it pre-provisioned compute wins. The infrastructure
is described in Terraform, k6 scripts and the analysis pipeline
reproduce every figure with a single \texttt{make} target, and the
artifact repository documents the migration from the AWS Academy
Learner Lab (deactivated mid-experiment) to a personal AWS account.
\end{abstract}
